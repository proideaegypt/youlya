## What Changed
- Added clear handoff-stop warning state in `/dashboard/conversations` and `/dashboard/inbox` when `ai_paused=true` or `status=handoff`: `الذكاء الاصطناعي متوقف لهذه المحادثة`.
- Added explicit recovery action button `إرجاع المحادثة للذكاء الاصطناعي` in both pages with confirmation prompt and success/error toasts.
- Added phone/conversation search UX input with placeholder `ابحث برقم العميل أو المحادثة...` on conversations and inbox.
- Updated key UI labels:
  - `Handoff` → `تحويل بشري`
  - `AI` → `ذكاء اصطناعي`
  - `AI متوقف` → `الذكاء الاصطناعي متوقف`
  - `مركز التحويل` → `مركز التحويل البشري`
- Hardened `POST /api/dashboard/conversations/[id]/return-to-ai`:
  - authenticated dashboard user required
  - authorized role required (`super_admin`/`owner`/`admin`/`customer_service`/`moderator`)
  - calls `returnToAI(conversationId, actor)`
  - returns safe JSON with state (`aiPaused`, `status`)
- Strengthened `returnToAI` service behavior to resolve all open handoff tickets for the conversation, clear AI pause state, and restore active AI conversation status.
- Implemented safe admin fixer flow:
  - `scripts/fix-stuck-handoff.mjs` wrapper
  - `scripts/fix-stuck-handoff.ts` logic using explicit `--conversation-id` or `--customer-id`, dry-run by default, `--confirm` required to mutate, masked identifiers, before/after verification (`ai_paused`, `status`, open handoff count), and `returnToAI` call.
- Added package script: `admin:fix-stuck-handoff`.
- Added/updated tests:
  - API unauthorized/authorized return-to-ai route behavior
  - UI presence of return-to-ai warning/action when conversation is paused
  - existing returnToAI unit tests continue to verify paused-state recovery and no-open-ticket case.
